# 04 — Database Model

This document describes the persistence layer: PostgreSQL tables,
enums, constraints, triggers, indexes, and how JPA entities map onto
them. For the conceptual model see [03-domain-model.md](./03-domain-model.md).

## Authoritative artifact

The canonical PostgreSQL schema lives at
`src/main/resources/db/database.sql` and is mirrored byte-for-byte at
`docs/database/schema.sql` for reference.

**Schema management is manual.** `dev` and `production` use
`spring.jpa.hibernate.ddl-auto=validate`: Hibernate validates the
schema on startup and refuses to mutate it. The `test` profile uses
`ddl-auto=none` with H2 in PostgreSQL compatibility mode (see below).

There is no Flyway or Liquibase configuration in this repository.

## Stack

- PostgreSQL 13+ in `dev` and `production`.
- Hibernate 6 via Spring Data JPA.
- PostgreSQL named enum types bound through
  `@JdbcTypeCode(SqlTypes.NAMED_ENUM)`.
- H2 in PostgreSQL compatibility mode in `test`. H2 cannot fully
  validate PostgreSQL-specific named enums, generated columns, and
  the `bone_catalog_component` view. Persistence tests for those
  features should run against real PostgreSQL — see
  [11-roadmap.md](./11-roadmap.md).

`culture_feature` (`culture_feature/CultureFeature.java`):

- `culture_feature_id` PK
- `culture_id` FK → `culture` ON DELETE CASCADE
- `feature` TEXT NOT NULL
- `updated_at` TIMESTAMPTZ (trigger)

### `archaeological_context`

Entity: `archaeological_context/ArchaeologicalContext.java`.

- `archaeological_context_id` PK
- `site_id` FK → `site` ON DELETE CASCADE (required)
- `stratigraphic_context` TEXT NOT NULL
- `culture_id` FK → `culture` ON DELETE SET NULL (nullable)
- `updated_at` TIMESTAMPTZ (trigger)

### `individual`

Entity: `individual/Individual.java`.

- `individual_id` PK
- `archaeological_context_id` FK → `archaeological_context`
  ON DELETE CASCADE (required)
- `individual_name` TEXT (nullable)
- `individual_type` `individual_type_enum` NOT NULL
- `mni` TEXT NOT NULL DEFAULT `'1'`
- `mni_statistical` INTEGER **GENERATED ALWAYS AS** (see below)
  **STORED**
- `age_at_death_text` TEXT
- `age_at_death_min`, `age_at_death_max` INTEGER
- `age_unit` `age_unit_enum`
- `age_class_main` `age_class_main_enum` NOT NULL
- `age_class_subcategory` `age_class_subcategory_enum`
- `sex` `sex_enum` NOT NULL
- `sex_certain` BOOLEAN NOT NULL
- `updated_at` TIMESTAMPTZ (trigger)

Constraints:

- `chk_mni_format`: `mni = '?' OR mni ~ '^[1-9][0-9]*$'`.
- `chk_age_range`: if both bounds present, `min <= max`.
- `chk_individual_type_mni_consistency`:
  - `INDIVIDUAL` requires `mni = '1'`;
  - `MIXED_INDIVIDUALS` requires `mni <> '1'`;
  - `UNASSIGNED_REMAINS` requires nothing extra.

Generated column `mni_statistical`:

```sql
mni_statistical INTEGER GENERATED ALWAYS AS (
  CASE
    WHEN mni ~ '^[1-9][0-9]*$' THEN mni::INTEGER
    WHEN mni = '?' THEN 1
    ELSE 1
  END
) STORED
```

JPA maps it with `insertable = false, updatable = false`. Never write
to it from code.

### `bone_catalog`

Entity: `bone_catalog/BoneCatalog.java`.

- `bone_catalog_id` PK — **seed data must use stable explicit IDs**
  because the `bone_catalog_component` view references fixed ids.
- `bone_catalog_name` TEXT NOT NULL UNIQUE
- `skeleton_main` `skeleton_main_enum`
- `skeleton_region` `skeleton_region_enum`
- `bone_category` `bone_category_enum` NOT NULL
- `term_type` `bone_term_type_enum` NOT NULL DEFAULT `'ATOMIC'`
- `parent_bone_catalog_id` FK → `bone_catalog` ON DELETE SET NULL

### `bone_catalog_component` (view)

A `CREATE OR REPLACE VIEW` defined with hard-coded
`(bone_catalog_id, component_bone_catalog_id, component_quantity)`
tuples. It expresses **anatomical composition** of grouped terms
(e.g. `cranium → neurocranium + splanchnocranium`,
`neurocranium → frontal + occipital + parietal×2 + sphenoid +
temporal×2 + ethmoid`, `coxal → ilium + ischium + pubis`).

Implementation rules:

- Treat it as read-only.
- Do not add table constraints. PostgreSQL does not allow them on
  views.
- Do not let Hibernate manage DDL for it. Use a subselect-style
  mapping only (it is not currently mapped as an `@Entity` with FK
  generation).
- If catalog IDs change, the view becomes wrong. Seed data must
  preserve the explicit IDs used inside the view definition.

### `bone`

Entity: `bone/Bone.java`.

- `bone_id` PK
- `individual_id` FK → `individual` ON DELETE CASCADE (required)
- `bone_catalog_id` FK → `bone_catalog` **ON DELETE RESTRICT**
  (required)
- `bone_source` TEXT NOT NULL
- `specimen_name`, `repository`, `year_discovery` TEXT
- `bone_processing` BOOLEAN
- `bone_quantity_source` TEXT
- `bone_quantity_min` INTEGER NOT NULL — CHECK `>= 1`
- `laterality` `laterality_enum`
- tooth/vertebra/phalanx/rib qualifier fields and their `_confidence`
  booleans
- `hand_foot_bone_segment` `hand_foot_bone_segment_enum`
- `bone_details` TEXT
- `updated_at` TIMESTAMPTZ (trigger)

### `skeleton`

Entity: `skeleton/Skeleton.java`.

- `skeleton_id` PK
- `individual_id` FK → `individual` ON DELETE CASCADE (required)
- `skeleton_category` `skeleton_category_enum` NOT NULL
- `preservation_index` NUMERIC — CHECK `0..100`
- `specimen_name`, `repository`, `year_discovery` TEXT
- `bone_processing` BOOLEAN
- `description` TEXT
- `updated_at` TIMESTAMPTZ (trigger)

### `dating_technique`

Entity: `dating_technique/DatingTechnique.java`.

- `dating_technique_id` PK
- `dating_technique_name` TEXT NOT NULL UNIQUE
- `description` TEXT
- `updated_at` TIMESTAMPTZ (trigger)

### `dated_sample`

Entity: `dated_sample/DatedSample.java`.

- `dated_sample_id` PK
- `sample_origin` `sample_origin_enum` NOT NULL
- `archaeological_context_id` INTEGER NULL FK
- `bone_id` INTEGER NULL FK
- `skeleton_id` INTEGER NULL FK
- `material` TEXT NOT NULL
- `dating_type` `dating_type_enum` NOT NULL
- `updated_at` TIMESTAMPTZ (trigger)

Constraints (also enforced in `DatedSampleService.validate`):

- exactly one of the three origin FKs is non-null;
- the non-null FK matches `sample_origin`;
- `chk_dating_type_origin_consistency`:
  - `CONTEXT` → `dating_type IN ('INDIRECT', 'UNCERTAIN')`;
  - `BONE` or `SKELETON` → `dating_type IN ('DIRECT', 'UNCERTAIN')`.

> Note: the `dated_sample` FK columns are declared in the schema
> without explicit `ON DELETE` modifiers. The other FKs in the schema
> file use the standard PostgreSQL default of `NO ACTION` unless
> explicitly stated; `archaeological_context_id`, `bone_id`, and
> `skeleton_id` on `dated_sample` therefore do **not** cascade from
> their parents — `Requires code verification` if exact behavior
> matters for a given migration.

### `dating_result`

Entity: `dating_result/DatingResult.java`.

- `dating_result_id` PK
- `dated_sample_id` FK → `dated_sample` ON DELETE CASCADE
- `dating_technique_id` FK → `dating_technique` **ON DELETE RESTRICT**
- `dates_bp_uncal` NUMERIC
- `dates_range` NUMERIC
- `notes` TEXT
- `updated_at` TIMESTAMPTZ (trigger)

### `bibliographic_reference` and bridges

`bibliographic_reference` (`reference/Reference.java`):

- `bibliographic_reference_id` PK
- `authors`, `year`, `title`, `journal` NOT NULL
- `volume`, `issue`, `pages`, `doi` (optional)
- `updated_at` TIMESTAMPTZ (trigger)

`site_bibliographic_reference`: composite PK
`(site_id, bibliographic_reference_id)`, both FKs ON DELETE CASCADE.

`archaeological_context_bibliographic_reference`: composite PK
`(archaeological_context_id, bibliographic_reference_id)`, both FKs
ON DELETE CASCADE.

### `funerary_context` and bridge

`funerary_context` (`funerary_context/FuneraryContext.java`):

- `funerary_context_id` PK
- `archaeological_context_id` FK → `archaeological_context`
  ON DELETE CASCADE (required)
- `burial_context` `burial_context_enum` NOT NULL DEFAULT `'UNCERTAIN'`
- `burial_type` `burial_type_enum` (nullable; constrained against
  `burial_context`)
- `ochre_presence`, `grave_goods` BOOLEAN
- `notes` TEXT
- `updated_at` TIMESTAMPTZ (trigger)

`chk_funerary_context_type`: `burial_context = 'NO'` ⇒
`burial_type IS NULL`; `burial_context = 'YES'` ⇒
`burial_type IS NOT NULL`; `POSSIBLE`/`UNCERTAIN` allow either.

`funerary_context_individual`: composite PK
`(funerary_context_id, individual_id)`, both FKs ON DELETE CASCADE.

## Triggers and timestamps

A shared trigger function maintains `updated_at`:

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS trigger AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Tables with this trigger: `site`, `archaeological_context`,
`individual`, `bone`, `bibliographic_reference`, `skeleton`,
`culture`, `culture_feature`, `dating_technique`, `dated_sample`,
`dating_result`, `funerary_context`. JPA maps `updated_at` as
`insertable = false, updatable = false`.

Bridge tables and the `bone_catalog` table do not have `updated_at`
and no trigger.

`app_user.created_at` and `refresh_token.created_at` use
`@CreationTimestamp` on the JPA side and SQL `DEFAULT
CURRENT_TIMESTAMP` on the DB side.

## Cascade behavior

Most archaeological-domain foreign keys use `ON DELETE CASCADE`:

- `site → archaeological_context → individual → bone / skeleton`;
- `archaeological_context → funerary_context`;
- `dated_sample → dating_result`;
- both bibliography bridge tables and
  `funerary_context_individual`.

Two foreign keys use `ON DELETE RESTRICT`:

- `bone.bone_catalog_id` — catalog terms cannot be deleted while
  bones reference them.
- `dating_result.dating_technique_id` — techniques cannot be deleted
  while results reference them.

`archaeological_context.culture_id` uses `ON DELETE SET NULL`.

Deleting a `Site` cascades through contexts, individuals, bones,
skeletons, funerary contexts, funerary memberships, and into dated
samples whose origin is one of those bones/skeletons (through their
respective parents). The backoffice should display strong delete
confirmations.

## Difference between entity model and API DTO model

The JPA entity graph is the persistence model. API consumers see a
narrower, flat projection:

- Response DTOs use `common/summary/*` records to avoid recursive
  serialization (e.g. `IndividualResponse` exposes an
  `ArchaeologicalContextSummaryResponse`, not a full
  `ArchaeologicalContextResponse`).
- API DTOs use camelCase; columns use snake_case.
- The API resource path `/api/references` exposes the
  `bibliographic_reference` table.
- `mni_statistical` is read-only in the API (never accepted in
  create/update requests).
- `updated_at` is read-only in the API.

See [05-api-contract.md](./05-api-contract.md) for the full DTO map
and [02-architecture.md](./02-architecture.md) for the mapper layer.
