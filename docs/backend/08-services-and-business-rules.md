# 08 — Services and Business Rules

This document describes what each `@Service` class does and lists the
non-obvious invariants future agents must preserve. Validation
rules are also cross-referenced in [07-validation-and-errors.md](./07-validation-and-errors.md).

## Conventions

- Services are `@Service` beans annotated `@Transactional`
  (read-only on read paths, default on writes).
- Services delegate persistence to Spring Data repositories.
- Services own the entity ↔ DTO mapping via static `Mapper` classes;
  detail responses are produced *inside* the transaction so that
  `@EntityGraph` detail queries can resolve nested data.
- Many-to-many association mutations replace the full set when the
  corresponding id list is defined in a PATCH request.

## SiteService (`site/`)

Responsibilities:

- `search(q, country, page, size, sort)` — paginated public list.
  `q` is trimmed; blank `q` is null. `country` is an exact match.
- `getDistinctCountries()` — returns the deduplicated list of
  countries (used by `/api/sites/countries`).
- CRUD: `getById`, `create`, `update`, `delete`.
- Uniqueness on `siteName` is enforced by the database; conflicts
  return HTTP `409` with a refined message.

## CultureService (`culture/`)

- Search, CRUD, `cultureFeature` management through the
  `Culture.features` collection.
- Enforces `startBp >= endBp` at service level (matches DB CHECK).

## ArchaeologicalContextService (`archaeological_context/`)

- Filters: `q`, `siteId`, `cultureId`.
- Detail fetch uses
  `ArchaeologicalContextRepository.findDetailById`, which eagerly
  loads `site`, `culture`, `culture.features`, and `references`.
- `update(...)` replaces `referenceIds` (many-to-many to
  `bibliographic_reference`) when the field is defined in the PATCH
  request.

## IndividualService (`individual/`)

Responsibilities:

- `search(...)` returns full `IndividualResponse` records via a
  two-step query: paginated id query then
  `findDetailsByIndividualIdIn(...)` with `@EntityGraph` to populate
  context/site/culture/bones/skeletons/funerary contexts in one
  follow-up.
- `searchSummary(...)` powers `GET /api/individuals/list` — it
  returns `IndividualSummaryDto` directly, projected from the
  database (no entity hydration).
- CRUD: `getById`, `create`, `update`, `delete`.

Allowed sort fields:

```
individualId, individualName, individualType, ageAtDeathMin,
ageAtDeathMax, ageClassMain, sex, updatedAt
```

Default sort: `individualId,asc`.

### Validation (`validateIndividual`)

- non-null `individualType`, `ageClassMain`, `sex`, `sexCertain`;
- `ageAtDeathMin <= ageAtDeathMax` when both present;
- `mni` matches `^\?$|^[1-9][0-9]*$`;
- `INDIVIDUAL` ↔ `mni = '1'`;
- `MIXED_INDIVIDUALS` ↔ `mni != '1'`;
- `UNASSIGNED_REMAINS` allows any valid mni.

`mni_statistical` is generated by the database; the service must
never write to it. The JPA entity declares it
`insertable = false, updatable = false`.

## BoneService and AdminBoneService (`bone/`)

- Public `BoneService`: paginated search with `q`, `boneCatalogId`,
  `individualId`; CRUD.
- `AdminBoneService` powers `/api/admin/bones` with an admin list
  projection (`AdminBoneListResponse`).
- `BoneMapper` rejects blank `boneSource` and validates
  `boneQuantityMin >= 1`.
- `boneCatalogId` reference is required; `ON DELETE RESTRICT` on the
  FK means a catalog entry cannot be deleted while bones reference
  it.

Curatorial fields (`specimenName`, `repository`, `yearDiscovery`,
`boneProcessing`) live on `Bone` directly — they are intentionally
not normalized into a separate `specimen` table.

## SkeletonService (`skeleton/`)

- Paginated search with `q`, `individualId`, `skeletonCategory`;
  CRUD.
- Validates `preservationIndex ∈ [0, 100]` when present.
- Curatorial fields live on `Skeleton` directly (same rationale as
  `Bone`).

`Skeleton` is a whole-skeleton preservation record. It is not a
collection of bones. Do not collapse it into `Bone`.

## BoneCatalogService (`bone_catalog/`)

- Search, hierarchical browsing through `parentBoneCatalogId`, and
  CRUD on the controlled vocabulary.
- The view `bone_catalog_component` is read-only and exposed only as
  a Hibernate subselect via `BoneCatalogComponent` / `BoneCatalogComponentRepository`.
- Seed data must use **stable explicit `bone_catalog_id` values**,
  because the view's grouped→component mappings are hard-coded.
  Reassigning catalog ids breaks the view.
- Do not use `component_quantity` as observed quantity (that's
  `bone.bone_quantity_min`).

## DatedSampleService (`dated_sample/`)

- Search filters: `sampleOrigin`, `datingType`,
  `archaeologicalContextId`, `boneId`, `skeletonId`.
- Detail fetch uses
  `DatedSampleRepository.findDetailById`, which eagerly loads
  `archaeologicalContext` (with site + culture), `bone` (with
  catalog and individual), `skeleton` (with individual),
  `datingResults`, and `datingResults.datingTechnique`.
- Service-level invariants (`validate`):
  - exactly one of `archaeologicalContext`, `bone`, `skeleton` is
    non-null;
  - `sampleOrigin` matches that non-null reference;
  - `CONTEXT` cannot have `datingType = DIRECT`;
  - `BONE` and `SKELETON` cannot have `datingType = INDIRECT`.

On PATCH, the service merges the current origin ids with the
defined ones from the update DTO and revalidates the result, so a
client can switch the origin in a single call.

## DatingResultService (`dating_result/`)

- Filters: `datedSampleId`, `datingTechniqueId`.
- Requires both `datedSampleId` and `datingTechniqueId` on create.
- Technique deletion is restricted by `ON DELETE RESTRICT` in the
  schema; in-use techniques cannot be removed.

## DatingTechniqueService (`dating_technique/`)

- Vocabulary table. `datingTechniqueName` is required and unique.
- Embed response (`DatingTechniqueEmbedResponse`) is used inside
  `DatingResultResponse` to avoid recursive nesting.

## FuneraryContextService (`funerary_context/`)

- Filters: `archaeologicalContextId`, `burialContext`.
- CRUD plus member-set replacement.
- Validates `burialContext`/`burialType` consistency:
  - `NO` ⇒ `burialType` must be null;
  - `YES` ⇒ `burialType` must be non-null.
- PATCH with a defined `individualIds` list **replaces** the
  membership through `funerary_context_individual`.

## ReferenceService (`reference/`)

- Search by `q` across bibliographic fields (authors, year, title,
  journal, doi). The exact JPQL is defined in
  `ReferenceRepository`.
- CRUD; bridge collections to `Site` and `ArchaeologicalContext` are
  managed from the parent side (PATCH on `Site`/`ArchaeologicalContext`),
  not from `Reference` itself.

## StatsService (`stats/`)

- `getHomeStats()` — single-pass aggregates:
  `IndividualRepository.sumMniStatistical()`,
  `BoneRepository.sumBoneQuantityMin()`, plus row counts for `site`
  and `skeleton`. Returns `HomeStatsResponse` with the canonical
  `lastUpdatedAt` (max `updated_at` across the public domain
  tables; null when every relevant table is empty).
- `getMapTimelineStats()` — composes three projection queries:
  - `SiteRepository.findMapTimelineSiteRows()`
  - `ArchaeologicalContextRepository.findMapTimelineUnitRows()`
  - `CultureRepository.findMapTimelineCultureRows()`
  - then folds per-site aggregates (`totalMni`, `individualCount`,
    `boneCount`, `skeletonCount`, `datedSampleCount`,
    `dominantCultureId`, `cultureIds`) over the unit rows.

## AdminDashboardStatsService (`stats/`)

- Returns `AdminDashboardStatsResponse` with row counts for `site`,
  `archaeological_context`, `individual`, `bone`, `bone_catalog`,
  `skeleton`, `bibliographic_reference`, `culture`,
  `funerary_context`, `dating_technique`, `dated_sample`,
  `dating_result`, `app_user`.
- Counts are simple `JpaRepository.count()` calls.

## BoneSiteSearchService (`bone_site_search/`)

- Builds a `BoneSiteSearchCriteria` from query params and delegates
  to the custom `BoneSiteSearchRepository` (impl in
  `BoneSiteSearchRepositoryImpl`).
- The repository issues a **single JPQL constructor projection**
  joining `Bone → BoneCatalog → Individual → ArchaeologicalContext →
  Site (LEFT JOIN Culture)` with `SIZE(b.datedSamples)` per match.
- Aggregation, sorting, and pagination run in Java on the flat row
  list inside the service.
- Repository-level `setMaxResults(app.search.bone-site.max-match-rows)`
  caps the broad-match budget.
- The controller caps page size at
  `app.search.bone-site.max-page-size`.

## DatasetExportService (`dataset/`)

- Builds the ZIP archive returned by `GET /api/dataset/download`.
- Writes UTF-8 CSV entries via `CsvWriter`.
- Caches the archive bytes for
  `app.dataset.export.cache-ttl-seconds`.
- Enforces `app.dataset.export.max-bytes` as a hard upper bound.
- Excludes authentication tables and the
  `bone_catalog_component` view.
- Logs `dataset_download` audit events through
  `SecurityAuditService`.

## UserService (`user/`)

- Admin-only CRUD on `AppUser`; self-service `getByUsername` and
  `changePassword`.
- `changePassword` revokes refresh tokens on success (the new
  password should invalidate prior sessions). *Requires code
  verification — `UserService.changePassword` calls into
  `RefreshTokenService`*; check the current implementation if the
  exact revocation list matters.
- Password hashing via `PasswordConfig.passwordEncoder()` (BCrypt).
- Username collisions raise `DataIntegrityViolationException`,
  mapped to HTTP `409` with a refined message.

## RefreshTokenService (`refresh_token/`)

- `storeNew` — persists hashed refresh tokens with family,
  per-token expiry, and family expiry.
- `consumeForRotationOrThrow` — atomic conditional update to consume
  the current token. Zero rows updated implies replay / invalid /
  already-rotated; the controller revokes the family.
- `linkReplacement` — writes `replaced_by_token_id` on the old row.
- `revokeFamily` — revokes all active tokens in a family.
- `RefreshTokenCleanupJob` periodically deletes expired rows.

`TokenHashService` produces the deterministic hash stored in
`refresh_token.token_hash`; the raw token never persists. Detailed
flow lives in [06-security-and-auth.md](./06-security-and-auth.md).

## Cross-cutting helpers

- `common/util/MapperUtils` — `JsonNullable` checks, string
  normalization, required-non-blank assertions.
- `common/util/PaginationUtils` — `Pageable` parsing with an
  allow-list of sort fields and page-size cap (`MAX_PAGE_SIZE =
  100`).
- `common/util/EntityFetchUtils` — small helpers for repository
  fetch-or-throw patterns.
- `common/summary/SummaryMappers` — single source for the flat
  summary DTOs used inside response records.

## Non-obvious behaviors to preserve

- `mni_statistical` is **DB-generated**; never write it from Java
  and never accept it in create/update requests.
- `updated_at` is **trigger-maintained**; never set it from Java.
- `bone_catalog_component` is a view with hard-coded ids; do not
  regenerate catalog ids without updating the view.
- Many-to-many PATCH operations replace the full set. If you need
  add-one / remove-one semantics, add new endpoints instead of
  redefining the existing PATCH behavior.
- `IndividualService.search` runs the count query and the id query
  with a paged sort, then re-loads the rows via an entity-graph
  detail query. This avoids N+1 fan-out across nested associations
  but means total query count is `count + ids + detail`. Preserve
  this two-step shape unless you replace it with a single
  constructor projection.
- `BoneSiteSearchService` returns aggregated rows produced in-memory
  from a single JPQL projection — do not refactor it into per-row
  follow-up queries.
- `StatsService.getMapTimelineStats` runs three projection queries.
  The frontend `MapTimelineStatsService` uses `shareReplay`, so a
  single client typically pays for the round-trip once. Backend
  changes that increase the response size or change the row shape
  will affect public web performance.
