# 10 — Roadmap

This document lists what is **confirmed missing** in the current codebase, and what is **speculative** for a future direction. Confirmed items are derived from the active code; speculative items come from earlier exploration documents that have otherwise been absorbed.

> "Confirmed" means: the gap is currently observable in `projects/web`, `projects/backoffice`, or `04-backend-api-contract.md`. "Speculative" means: a worthwhile direction has been considered but no implementation has started.

### Backoffice

| Item | Where | Notes |
|---|---|---|
| No lightweight admin list projections for several domains | backend | `/api/admin/sites`, `/api/admin/individuals`, `/api/admin/bones` exist. Cultures, references, skeletons, funerary contexts, dating still call the public `/api/<entity>` endpoints, which carry heavy nested payloads. |

### API / backend wishlist

| Item | Why |
|---|---|
| `country` query parameter on `/api/individuals` | The public Individuals list cannot offer a global country filter today. |
| `cultureId` query parameter on `/api/individuals` | Same as country. |
| `cultureId` query parameter on `/api/archaeological-contexts` | Already exists in the v2 contract (`ArchaeologicalContextService.search` passes it). Keep documenting. |
| `lastUpdatedAt` field on `HomeStats` | The frontend is already wired to display it when present; the line is currently hidden because the backend omits it. |
| `GET /api/individuals/{id}/bundle` (or `?expand=`) | Collapse the Phase 3 dating fan-out into one request. |
| `Cache-Control: public, max-age=…` headers on read-only endpoints | Required to make the hydration transfer cache and edge cache meaningful. |
| `Content-Encoding: br` / `gzip` on JSON | Largest endpoints (130 KB stats, 2.5 MB individuals, 110 KB bone-site-search) ship uncompressed. |
| `GET /api/stats/analysis` (or per-chart aggregations) | Allows `/analysis` to stop sweeping the full individuals collection. |
| Public/admin classification for `/api/dating-techniques`, `/api/dating-results`, `/api/users` (post-v2) | `Requires code verification` against the current Spring `SecurityConfig`. |

## Speculative directions

These ideas appeared in older exploration documents and have not been committed to. They are recorded so future agents see them, not as scheduled work.

| Direction | Sketch |
|---|---|
| Switch parametric routes to `Prerender` with `getPrerenderParams` | Replace `RenderMode.Server` on `/sites/:id` and `/individuals/:id`. Viable today because the datasets are small (248 sites, 773 individuals). Trade-off: longer build, but no SSR cost per visit. |
| Per-site media surface | A media presentation (images, drawings) for sites and / or individual records. No backend asset endpoint today; requires schema and storage work. |
| Standalone funerary-context / multi-burial page | A dedicated page for multi-individual burials (Barma Grande, Dolní Věstonice triple burials). Backend supports the data; no frontend route exists. |
| Better separation between Inter / JetBrains Mono | The DEEP TIME design intent says data uses JetBrains Mono. `dark-theme.css` currently aliases all `--dt-font-*` slots to Space Grotesk. A future pass can load Inter + JetBrains Mono and un-alias the tokens. |
| Mini-timeline strip on every record page | Stated as a DEEP TIME principle. Partial implementation; not consistently present on Site Details and Individual Details. |
| Switch to a more scholarly icon set | Material Symbols signal "consumer app". The replacement is not chosen. |
| Animated chronological marker entrance on the map | Considered in the DEEP FIELD exploration; rejected for SSR risk and color-competition with culture palette. Could be revisited as an optional view. |

## Out of scope

Things that should not appear in any roadmap pulled from this repository:

- Anything that changes database schema. Schema changes belong in the backend repo and the `/docs/database/` reference.
- Anything that changes the Spring backend's business logic or response shape. The frontend can request the change, but the implementation lives elsewhere.
- Cross-app shared libraries between `projects/web` and `projects/backoffice`. The duplication is intentional today.
